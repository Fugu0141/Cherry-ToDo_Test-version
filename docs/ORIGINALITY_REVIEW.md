# Originality Review

## Purpose

Cherry-ToDo is being prepared as an open-source project. Before wider public development, the project should clearly document its inspiration, originality, and areas that must avoid being too close to existing products.

This document is not legal advice. It is a practical review checklist for respectful open-source development.

---

## Background

Cherry-ToDo was influenced by the idea of organizing tasks as a flow instead of only as a simple list.

The project was originally developed as a personal prototype under `Fugu0141.github.io/ToDo` and later moved to this standalone repository for open-source development.

The project should not clone or reproduce any existing service.

---

## Development stance

Cherry-ToDo may be inspired by general ideas such as:

- task flow
- branches
- task blocks
- visual organization
- date-aware task management

However, Cherry-ToDo should avoid copying:

- exact UI layout from another product
- distinctive visual style from another product
- product-specific wording
- onboarding text or marketing copy
- pricing or business positioning
- screenshots, icons, illustrations, or assets
- code from another product

---

## Cherry-ToDo's own direction

Cherry-ToDo should develop around its own design ideas:

- root tasks as projects, tags, or large categories
- child tasks as actual actions
- board view for creating task flow
- list view for execution
- unscheduled tasks as a first-class state
- schedule model with none / date / datetime
- same-day subflow display
- mobile-specific UI instead of desktop shrinking
- open-source specification-driven development

---

## Public explanation draft

A safe public explanation can be:

```text
Cherry-ToDo is a personal open-source task management project inspired by the idea of organizing tasks as flows instead of simple lists.

It is not intended to clone or reproduce any existing service. The project focuses on root tasks as project labels, branch-based task flow, schedule-aware layout, unscheduled tasks, and open-source development.
```

---

## Review checklist before release

Before public promotion, check:

- Does the app look like a separate product at a glance?
- Are the app name, logo, colors, and wording original enough?
- Does the interaction flow avoid directly recreating another product's distinctive behavior?
- Are all assets made by the project or properly licensed?
- Does the README describe Cherry-ToDo's own direction?
- Are credits or inspiration notes phrased carefully and respectfully?
- Is the app not marketed as a replacement or free clone of another individual developer's paid product?

---

## Risk levels

### Higher risk

- copying another product's screen structure too closely
- using similar wording or feature names
- making the app visually look like the same product
- presenting the project as a free alternative to a specific paid product

### Medium risk

- similar feature categories
- similar broad interaction ideas
- similar task-flow concepts

### Lower risk

- being a todo app
- supporting dates
- supporting task hierarchy
- supporting drag-and-drop
- supporting visual boards

---

## Practical rule

When in doubt, make Cherry-ToDo more clearly itself.

Change wording, colors, layout, icons, and interaction details until it is easy to explain the project as an independent design.
